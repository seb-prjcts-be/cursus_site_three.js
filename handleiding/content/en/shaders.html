Shaders are small programs that run directly on the graphics card: one for the shape (vertex shader), one for the colour of each pixel (fragment shader). If you know shaders from p5, this is familiar territory in a new outfit.

What you will learn
  • ShaderMaterial is a material whose GLSL code you write yourself
  • Uniforms are variables you pass in from JavaScript (time, mouse, colour)
  • three.js gives you position, normal and uv for free in the vertex shader
  • Same GLSL language as p5 shaders, but without the setUniform fuss: uniforms are an object

The smallest shader

A ShaderMaterial has two pieces of GLSL and a uniforms object. This is the minimum that shows something:

const material = new THREE.ShaderMaterial({
  uniforms: {
    uTime: { value: 0 }          // to be updated from JS
  },
  vertexShader: `
    varying vec2 vUv;
    void main() {
      vUv = uv;                  // pass uv on to the fragment shader
      gl_Position = projectionMatrix * modelViewMatrix * vec4(position, 1.0);
    }
  `,
  fragmentShader: `
    uniform float uTime;
    varying vec2 vUv;
    void main() {
      gl_FragColor = vec4(vUv, abs(sin(uTime)), 1.0);
    }
  `
});

In p5, loadShader() did this work and you passed uniforms with setUniform. Here the uniforms object is the bridge: material.uniforms.uTime.value = t in your loop, done.

The line with projectionMatrix * modelViewMatrix is the standard line that projects 3D points onto the screen - you copy it literally like this into almost every vertex shader.

Start by playing in the fragment shader: replace the pink vec3 with a colour of your own, or multiply uTime by 10. Fragment mistakes are safe and the effect is immediately visible. You bend vertex shaders after that.

Watch out

GLSL is strict: 1 is an int and 1.0 a float, and you cannot just mix them. The error message appears in your browser console. Nine out of ten shader bugs are a forgotten .0.

When do you reach for a shader?

  • Effects that have to happen per pixel or per vertex: waves, gradients, deformations, dissolves.
  • When thousands of objects need the same effect - the GPU does it in one go.
  • Not for things a standard material can already do: always start with MeshStandardMaterial and only escalate to a shader when you really have to.
Did you know

Every built-in three.js material is a shader - MeshStandardMaterial is thousands of lines of generated GLSL. So with ShaderMaterial you are not writing an "extra", you are taking the pen from the library itself. The newest three.js versions also have TSL (Three Shading Language) to write shaders in JavaScript for WebGPU - comparable to p5.strands in p5 2.x.
